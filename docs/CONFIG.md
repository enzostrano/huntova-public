# Configuration

Huntova splits configuration into **two stores** so secrets stay out
of disk-readable text:

| What                          | Where                                              |
|-------------------------------|----------------------------------------------------|
| **API keys + secrets**        | OS keychain (macOS Keychain / Windows Credential Manager / Linux Secret Service) — falls back to `~/.config/huntova/secrets.enc` (Fernet-encrypted) on systems without a keyring |
| **Non-secret preferences**    | `~/.config/huntova/config.toml`                    |
| **Lead data + hunt history**  | `~/.local/share/huntova/db.sqlite`                 |
| **Plugin source files**       | `~/.config/huntova/plugins/*.py` (auto-discovered) |

Secrets are **never** written to `config.toml`, **never** committed
to git (the default `.gitignore` excludes the entire `huntova/`
config dir), and **never** appear in log output (`huntova config
show` redacts anything matching `*key*`, `*token*`, `*secret*`,
`*password*`).

## Operational commands

```bash
huntova config path                # prints the config.toml path
huntova config show                # pretty-prints config (secrets redacted)
huntova config get <key>           # prints one value
huntova config set <key> <value>   # writes one value (refuses to write secrets)
huntova config edit                # opens config.toml in $EDITOR
```

`huntova config set` deliberately refuses to write keys whose name
contains `password`, `secret`, or `_key` — those route through the
keychain-only path. Use `huntova onboard` to add or update an API
key.

## Default `config.toml` template

Generated by `huntova onboard` (or `huntova init`) on first run.

```toml
# Huntova configuration
# Non-secret settings only. API keys live in your OS keychain
# (or ~/.config/huntova/secrets.enc fallback).

# AI provider used for scoring + email drafting
preferred_provider = "gemini"

# Hunting defaults
[hunting]
default_countries = ["USA", "United Kingdom", "Germany", "France"]
max_leads_per_hunt = 25
playwright_deep_qualify = true

# Outreach (cold-email send) — credentials in keychain, not here
[outreach]
daily_send_cap = 50
default_dry_run = true   # require --no-dry-run to actually send

# Workspace
[workspace]
name = "default"

# Optional proxy (for SearXNG / web fetches)
[proxy]
http = ""
https = ""

# Telemetry — opt-in. See `huntova telemetry enable`.
```

## Environment variables

Env vars override anything in `config.toml`. Useful for CI / Docker /
ephemeral configs.

### AI providers (one is required)

| Variable                  | Purpose                                                        |
|---------------------------|----------------------------------------------------------------|
| `HV_GEMINI_KEY`           | Google AI Studio key (default provider)                        |
| `HV_ANTHROPIC_KEY`        | Anthropic Console key                                          |
| `HV_OPENAI_KEY`           | OpenAI Platform key                                            |
| `HV_OPENROUTER_KEY`       | OpenRouter key (200+ models, single API)                       |
| `HV_GROQ_KEY`             | Groq Cloud key (fastest inference)                             |
| `HV_DEEPSEEK_KEY`         | DeepSeek key                                                   |
| `HV_TOGETHER_KEY`         | Together AI key                                                |
| `HV_MISTRAL_KEY`          | Mistral La Plateforme key                                      |
| `HV_PERPLEXITY_KEY`       | Perplexity Sonar key                                           |
| `HV_OLLAMA_KEY`           | Optional auth for local Ollama (default: none)                 |
| `HV_LMSTUDIO_KEY`         | Optional auth for local LM Studio                              |
| `HV_LLAMAFILE_KEY`        | Optional auth for local llamafile                              |
| `HV_CUSTOM_KEY`           | Custom OpenAI-compatible endpoint key                          |
| `HV_CUSTOM_BASE_URL`      | Custom endpoint URL (e.g. `https://my-gateway/v1`)             |
| `HV_CUSTOM_MODEL`         | Custom endpoint model name                                     |
| `HV_AI_PROVIDER`          | Override `preferred_provider` (e.g. `ollama`)                  |
| `HV_<NAME>_MODEL`         | Override default model per provider                            |
| `HV_<NAME>_BASE_URL`      | Override default base URL per provider                         |

### Web search (Huntova hunts via SearXNG)

| Variable                  | Purpose                                                        |
|---------------------------|----------------------------------------------------------------|
| `SEARXNG_URL`             | Self-hosted SearXNG instance (recommended for hunts)           |

Public SearXNG instances often disable the JSON API to prevent abuse,
so the cleanest setup is to run your own:

```bash
docker run -d --restart=always --name=searxng -p 8888:8080 searxng/searxng
export SEARXNG_URL=http://localhost:8888
```

Or use the Huntova Cloud Search proxy (paid tier — `huntova cloud
token mint` if you have an admin token).

### Outreach (cold-email send via your SMTP)

| Variable          | Purpose                                                |
|-------------------|--------------------------------------------------------|
| `SMTP_HOST`       | SMTP server (e.g. `smtp.gmail.com`)                    |
| `SMTP_PORT`       | Port (default 587 STARTTLS)                            |
| `SMTP_USER`       | Login username                                         |
| `SMTP_PASSWORD`   | Login password / app password (lives in keychain)      |
| `SMTP_FROM_EMAIL` | From address (default `noreply@huntova.com`)           |
| `SMTP_FROM_NAME`  | From display name (default `Huntova`)                  |

Without these, `huntova outreach send` requires `--dry-run`. It'll
refuse to mail until you configure SMTP.

### Storage

| Variable          | Purpose                                                |
|-------------------|--------------------------------------------------------|
| `HUNTOVA_DB_PATH` | Override the SQLite path                               |
| `DATABASE_URL`    | If set (and points at Postgres), Huntova uses Postgres instead. Useful for cloud / multi-user deploys. |

### Operational

| Variable          | Purpose                                                |
|-------------------|--------------------------------------------------------|
| `APP_MODE`        | `local` (CLI default) / `cloud` (server default)       |
| `HV_PUBLIC_URL`   | Override the public URL used in OG cards / share links |
| `HV_SECRET_KEY`   | App secret for signing cookies (cloud mode)            |
| `HV_NO_BROWSER`   | If set to `1`, disables auto-open of the browser       |
| `HV_KILL_TRY`     | Emergency kill-switch for `/api/try` (returns 503)     |
| `HV_RECIPE_URL_BETA` | Enable the gated `/r/<slug>` recipe-sharing routes  |
| `HV_CLOUD_PROXY_BETA` | Enable the gated `/cloud-search/<token>/...` route |
| `HV_CLOUD_SEARXNG_URL`| Upstream SearXNG for the Cloud Proxy MVP           |
| `HV_ADMIN_TOKEN`  | Admin bearer token for `/api/admin/*` endpoints        |
| `HV_DEMO_AI_KEY`  | Server-side AI key for the public `/try` preview demo  |

### Telemetry (opt-in only)

| Variable          | Purpose                                                |
|-------------------|--------------------------------------------------------|
| `HV_METRICS_URL`  | Override the telemetry POST endpoint                   |

Telemetry is **off by default**. Run `huntova telemetry enable` to
opt in. Three events fire when enabled: `try_submit` (server-side),
`cli_init`, `cli_hunt`. No keys, no queries, no PII — just platform,
version, and counts.

## Where each setting takes effect

`huntova hunt` resolves provider keys in this priority order:

1. Per-call settings dict (used internally by the cloud server)
2. OS keychain via `secrets_store.get_secret(env_var)`
3. Process environment via `os.environ[env_var]`

This means you can safely export a key in your shell for a single
hunt without writing it to disk, and Huntova will pick it up.

## Editing under version control

If you want to keep your `config.toml` in git, the **non-secret**
fields are safe to commit:

- `preferred_provider`
- `[hunting].default_countries`, `max_leads_per_hunt`, `playwright_deep_qualify`
- `[outreach].daily_send_cap`, `default_dry_run`
- `[workspace].name`
- `[proxy].http`, `[proxy].https`

The secrets store and `~/.config/huntova/secrets.enc` are **never**
safe to commit; both are excluded from `huntova` configurations by
default.

## Verifying configuration

```bash
huntova doctor               # runs every health check
huntova status               # operational dashboard
huntova test-integrations    # probes each configured AI provider, SearXNG, Playwright, plugins, SMTP
```

`huntova doctor` is the canonical "is anything broken?" command and
exits non-zero if a critical integration fails. CI / launch gates
should rely on it.
